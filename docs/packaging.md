# Packaging notes

This is a single-binary egui/eframe app — there's no installer, asset bundle, or runtime
helper to ship. Everything you need is `pob-desktop` plus the `data/` directory next to
it (or one level up — the loader probes both).

## macOS

```bash
cargo build -p pob-desktop --release
# Result: target/release/pob-desktop
```

For a `.app` bundle, use `cargo bundle`:

```bash
cargo install cargo-bundle
cargo bundle --release -p pob-desktop
# Result: target/release/bundle/osx/Path of Building MK2.app
```

`cargo-bundle` reads metadata from `app/pob-desktop/Cargo.toml`. Add a `[package.metadata.bundle]`
block when you want a real bundle (icon, identifier, copyright). For ad-hoc local use,
the bare binary works fine.

Codesigning + notarisation are separate concerns:

```bash
codesign --deep --sign "Developer ID Application: NAME (TEAMID)" \
  target/release/bundle/osx/Path\ of\ Building\ MK2.app
xcrun notarytool submit ... # submit + staple per Apple docs
```

## Windows

From a native Rust toolchain on Windows:

```cmd
cargo build -p pob-desktop --release
```

Cross-compiling from another platform requires the `x86_64-pc-windows-msvc` target plus
the corresponding linker (msvc on Windows; `cargo-xwin` from elsewhere). The wgpu
backend uses DXGI by default on Windows, no extra setup needed.

For a signed installer, [WiX](https://wixtoolset.org/) or [Inno Setup](https://jrsoftware.org/isinfo.php)
both work. Bundle `pob-desktop.exe` plus a sibling `data/` directory.

## Linux

```bash
cargo build -p pob-desktop --release
```

eframe needs the usual X11/Wayland deps (`libxkbcommon`, `libwayland-client`,
`libxcursor`, `libxrandr`, `libxi`, `libgl`). On Debian/Ubuntu:

```bash
apt install libxkbcommon-dev libwayland-dev libxcursor-dev libxrandr-dev libxi-dev libgl1-mesa-dev
```

A `.deb` via `cargo-deb`, or an AppImage via `linuxdeploy`, are both viable. The binary
is statically-linked against Rust deps, dynamically-linked against libc + the X/Wayland
graphics stack.

## Updates / auto-update

Out of scope. The original PoB ships a manifest-driven updater that pulls patches each
time GGG releases a content patch. That makes sense when the engine and data move at
different cadences, but ours don't yet — we re-extract from the upstream PoB checkout
whenever it bumps a tree version. A user-facing update story can come later via
`cargo dist` or `tauri-updater`-style infrastructure.

## Data directory

The desktop binary searches `data/` relative to the working directory, then `../data`,
then `../../data`. For installer builds, ship `data/` next to the binary so the first
path matches.

`data/` is regenerated by `cargo run -p pob-extract --release`. The extractor needs a
sibling checkout of [PathOfBuildingCommunity/PathOfBuilding](https://github.com/PathOfBuildingCommunity/PathOfBuilding)
at `../PathOfBuilding/` (or specify with `--pob`). Extraction runs once and produces
`bases.json` (~870 KB), `gems.json` (~445 KB), `skills/*.json` (~5 MB), and
`trees/*.json` (~40 MB across 28 modern tree versions).

## Build storage

Where saved builds live depends on which build you're running.

**Desktop.** Saves land in a per-OS app-data folder:

- macOS: `~/Library/Application Support/PathOfBuildingMK2/Builds/`
- Linux: `$XDG_DATA_HOME/PathOfBuildingMK2/Builds/` (falls back to `~/.local/share/...`)
- Windows: `%APPDATA%\PathOfBuildingMK2\Builds\`

Each build is a `.mk2` (compact share-code) or `.xml` (PoB-compatible) file. One
level of subdirectories acts as categories ("Levelling/", "Bossing/", etc.).

**Wasm (browser).** No real filesystem to write into, so the Builds tab uses up to
three layers, in priority order:

1. **Connected folder** (Chromium-only, opt-in). If the browser exposes the
   File System Access API and the user clicks "Connect folder," the app reads/writes
   `.mk2` / `.xml` files directly in the chosen directory. The directory handle is
   persisted in IndexedDB so the connection survives reload (the browser may re-prompt
   for permission). Click "Disconnect folder" to fall back to layer 2.
2. **IndexedDB** (default). Saves the `.mk2` payload + metadata in a per-origin
   IndexedDB store (`pob_mk2_builds`). The list survives reload but is wiped by
   "Clear site data" / "Clear browsing data → Cookies and other site data."
3. **Manual download** (always). Every Save also triggers a browser download of the
   `.mk2` file so the user keeps a real file on disk regardless of the active layer.
   Use **Import file…** in the Builds tab (or `File → Open…`) to bring a saved file
   back in.

Storage quota errors (e.g. `QuotaExceededError`) surface a toast in the status bar
rather than failing silently — typically a sign to delete old saves or connect a
folder.
